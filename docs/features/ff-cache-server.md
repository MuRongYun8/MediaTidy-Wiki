# FF 缓存

FF 缓存用于复用媒体文件的 FF 信息，核心目标是两个：**快速整理**，以及**快速恢复库内媒体信息**。

MediaTidy 里现在有两类 FF 缓存：

| 类型 | 主要作用 | 解决的问题 |
|------|----------|------------|
| 整理 FF | 缓存 `ffprobe` 探测结果 | 自动整理、洗版、质量评分时不必反复探测同一个文件 |
| 神医 FF | 缓存 Emby 媒体源信息 | 重建 STRM 库、迁移媒体库、恢复播放信息时不用重新逐个读取 Emby 媒体源 |

简单理解：整理 FF 让 MediaTidy 更快判断“这个文件是什么质量”；神医 FF 让 MediaTidy 更快恢复“库里这个媒体该怎么播放、有什么音视频流”。

## 整理 FF 是什么

自动整理不只是改名和移动文件。为了判断版本质量、洗版优先级、命名标签和规则匹配，MediaTidy 需要读取视频文件里的技术信息，例如：

- 分辨率、宽高
- 视频编码、音频编码
- 声道、Atmos
- HDR / Dolby Vision / SDR
- 色深、帧率
- 字幕信息
- 码率、时长
- 文件扩展名和基础体积信息

这些信息通常来自 `ffprobe`。如果每次整理、洗版、重新扫描都重新跑一遍，云盘文件会很慢，也会增加挂载、下载直链和媒体服务器的压力。

整理 FF 缓存会把这些结果按文件 SHA1 保存起来。后续遇到同一个文件时，即使路径变了、目录换了、重新整理了一次，也可以直接复用已经探测过的结果。

## 整理 FF 能加速什么

整理 FF 主要加速这些流程：

| 场景 | 没有缓存 | 命中整理 FF 后 |
|------|----------|----------------|
| 首次大批量整理 | 每个视频都要探测媒体参数 | 已缓存文件直接跳过探测 |
| 洗版升级 | 反复读取候选版本质量信息 | 快速拿到分辨率、HDR、音频等评分依据 |
| 重新整理同一批文件 | 再次执行 `ffprobe` | 根据 SHA1 直接复用旧结果 |
| 多目录/多节点整理 | 每个节点各自探测 | 可共享同一份探测结果 |
| 规则调试后重跑 | 重复读取媒体技术参数 | 把时间花在规则和转移上，而不是重复探测 |

::: tip
整理 FF 不决定影片识别结果，影片识别仍然依赖文件名、目录、TMDB 和规则。它负责提供“这个视频文件本身的技术质量信息”。
:::

## 整理 FF 的工作流程

```mermaid
flowchart LR
  A["整理任务发现视频"] --> B["读取文件 SHA1"]
  B --> C["查询整理 FF 缓存"]
  C -- 命中 --> D["直接得到分辨率/编码/HDR/音频等信息"]
  C -- 未命中 --> E["执行 ffprobe 探测"]
  E --> F["写入整理 FF 缓存"]
  D --> G["质量评分 / 洗版判断 / 命名规则"]
  F --> G
```

只要 SHA1 一致，同一个媒体文件在不同目录、不同整理方案中都可以复用同一份整理 FF。

### 旧缓存什么时候需要重新 FFprobe

缓存命中不是要求“所有可能字段永远齐全”，而是只检查本次命名和评分规则真正需要的能力：

- 旧式 `audio_codec.priority` 和 `subtitle.priority` 只使用音频/字幕摘要，不要求完整轨道清单，也不会仅为缺少清单重新探测。
- 结构化轨道条件、`min_tracks`、`max_tracks`、`preferred_tracks` 才要求完整的 `audioTracks` 或 `subtitleTracks`。
- 音轨字段明确存在，或 `ProbeRevision >= 2`，表示音轨清单能力已满足；仅在字段缺失且 revision 低于 2 时补探。
- 字幕字段明确存在，或 `ProbeRevision >= 4`，表示字幕清单能力已满足；仅在字段缺失且 revision 低于 4 时补探。
- `ProbeRevision >= 2` 已覆盖普通固定全量字段。分辨率、编码、HDR、色深、帧率、字幕摘要或总码率为空时，表示该次完整探测没有取得值，不会因为字幕能力后来升级到 revision 4 就重复探测。
- 用户没有启用新的音轨或字幕轨规则时，不检查对应完整清单，因此不会让该用户的旧文件批量重新 FFprobe。

`ProbeRevision` 是缓存能力契约，不会被忽略。字段显式存在可以让较低 revision 的缓存提前满足能力；字段缺失时才使用 revision 判断它是“旧版没采集”还是“新版已采集但为空”。

一旦缓存确实不满足本次需求并执行真实 FFprobe，MediaTidy 会采集并保存固定完整载荷，避免今后逐字段重复读取文件。容器总码率、视频码率、音轨清单和字幕清单属于逐文件事实，不会使用某一集的系列代表结果代替其他集；远端预取也会查询每个文件的 SHA1。

## 神医 FF 是什么

神医 FF 面向 STRM 和 Emby 媒体库恢复。它缓存的不是 `ffprobe` 结果，而是 Emby 里与播放相关的媒体源信息，例如：

- `MediaSource`
- `MediaStreams`
- 视频流、音频流、字幕流
- Emby Item ID
- MediaSource ID
- 容器、路径、码率、编码等播放侧信息

这些信息对 STRM 库很重要。重建 STRM、迁移 Emby、重新生成库之后，如果要恢复媒体条目的播放信息、音视频流信息和媒体源信息，逐个重新读取会很慢。

神医 FF 会把这些媒体源信息缓存起来。后续需要恢复时，MediaTidy 可以按文件 SHA1 快速取回对应信息，减少重新扫库和重新读取媒体源的时间。

## 神医 FF 能加速什么

神医 FF 主要加速这些流程：

| 场景 | 没有缓存 | 命中神医 FF 后 |
|------|----------|----------------|
| STRM 库重建 | 需要重新读取大量 Emby 条目 | 快速恢复已有媒体源信息 |
| 媒体库迁移 | 新库需要重新补齐媒体流信息 | 按 SHA1 找回原来的媒体信息 |
| 神医批量预热 | 大量请求 Emby 获取媒体源 | 已缓存条目直接恢复 |
| 整理后入库 | 等 Emby 扫描后再读取媒体源 | 可延迟采集并写入神医 FF |
| 误删/重建 STRM | 播放信息需要重新生成 | 用缓存快速恢复库内媒体信息 |

::: tip
神医 FF 不是 STRM 文件内容缓存，也不是视频文件缓存。它缓存的是 Emby 已识别出来的媒体源和媒体流信息，用来让库恢复得更快。
:::

## 神医 FF 的工作流程

```mermaid
flowchart LR
  A["神医任务读取 Emby 条目"] --> B["获取文件 SHA1"]
  B --> C["查询本地 JSON / 本地缓存"]
  C -- 命中 --> H["恢复媒体源信息"]
  C -- 未命中 --> D["查询神医 FF 缓存"]
  D -- 命中 --> H
  D -- 未命中 --> E["从 Emby 提取 MediaSource / MediaStreams"]
  E --> F["写入神医 FF 缓存"]
  F --> H
  H --> G["快速恢复库内媒体信息"]
```

## 两者的区别

| 对比项 | 整理 FF | 神医 FF |
|--------|---------|---------|
| 面向流程 | 自动整理、洗版、质量评分 | STRM 神医、媒体库信息恢复 |
| 数据来源 | `ffprobe` 探测视频文件 | Emby 媒体源和媒体流信息 |
| 缓存内容 | 分辨率、编码、HDR、音频、字幕、码率、时长等 | MediaSource、MediaStreams、Item ID、MediaSource ID 等 |
| 主要收益 | 快速整理、减少重复探测 | 快速恢复库内媒体信息、减少重复读取 Emby |
| 典型使用 | 大批量整理、重复洗版、多节点整理 | STRM 库重建、Emby 迁移、神医预热 |

## 推荐用法

如果你经常做大批量整理、洗版或重跑整理方案，建议开启整理 FF。它能明显减少重复 `ffprobe`，尤其是云盘文件较多时。

如果你使用 STRM，并且会重建库、迁移库、批量恢复播放信息，建议开启神医 FF。它能让 MediaTidy 更快找回库内媒体源和媒体流信息。

两者可以独立使用，也可以同时使用。常见组合是：

1. 整理阶段使用整理 FF，加速质量判断和洗版。
2. 整理完成后等待 Emby 扫库。
3. 神医任务采集并缓存媒体源信息。
4. 下次重建或迁移时，通过神医 FF 快速恢复库内媒体信息。

## 配置入口

在「系统设置 → TMDB / 缓存」中可以配置两套 FF 缓存：

| 配置块 | 说明 |
|--------|------|
| 整理远程 FF 缓存 | 用于自动整理、洗版、质量评分 |
| 神医远程 FF 缓存 | 用于 STRM 神医和媒体库信息恢复 |

如果只是单机使用，本地 FF 缓存已经能带来一部分加速；如果有多台 MediaTidy、经常迁移库或需要共享缓存结果，再配置远程节点会更有价值。
