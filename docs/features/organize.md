# 自动整理

自动整理是 MediaTidy 的核心功能，通过 TMDB 识别媒体文件并按规范目录结构自动归档。

## 功能概览

- 基于 TMDB 自动识别电影和电视剧
- 按可自定义的命名规则重命名和归档
- 支持移动、复制、软链接、硬链接四种传输方式
- 伴随文件（字幕、NFO、海报）随主文件一起转移
- 洗版升级：自动检测更高质量版本并替换
- 未识别文件可转移到兜底目录
- 支持分类引擎按 TMDB 元数据自动分流到不同媒体库
- 按路径自动匹配 CD2 或本地 FUSE 整理模式

## 创建整理方案

进入「自动整理」页面，点击「新建方案」。每个方案是一组独立的配置，可以创建多个方案处理不同的源目录。

### 基本配置

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **方案名称** | `name` | 自定义名称，便于在列表中识别 |
| **启用** | `enabled` | 关闭后自动扫描跳过此方案，但手动触发仍可执行 |
| **源目录** | `sourceDir` | 待整理文件所在的目录（虚拟路径格式，如 `/我的115/downloads`） |
| **目标目录** | `destDir` | 整理后文件存放的目录（虚拟路径格式） |
| **未识别目录** | `unrecognizedDir` | TMDB 无法识别的文件转移到此目录（留空则跳过未识别文件） |
| **所属账号** | `accountId` | 关联的 115 账号 ID（用于路径解析和 API 调用） |

### 传输方式

| 值 | 说明 | 适用场景 |
|----|------|----------|
| `move` | 移动文件（源文件删除） | 常规整理，释放源目录空间 |
| `copy` | 复制文件（源文件保留） | 需要保留原始文件 |
| `symlink` | 创建符号链接 | 本地文件系统，节省空间 |
| `hardlink` | 创建硬链接 | 同一文件系统内，节省空间 |

### 扫描与文件操作

自动整理没有单独的「数据源」选项。MediaTidy 会根据源目录路径和[网盘配置](/config/drive#路由配置)自动决定整理方式：

- 115 路径映射和通用挂载点会走 CloudDrive2 / FUSE 对应的文件操作通道
- 已配置 FUSE 时，文件管理、整理、移动、复制等操作可以优先使用本地 FUSE
- 没有配置 FUSE 时，保持使用 CD2 gRPC 通道即可

::: tip 推荐
新手只需要先把「网盘配置 → 路由配置」设置好。已挂载 `/CloudNAS` 的环境推荐文件访问使用「本地 FUSE」；没有 FUSE 时保持默认 CD2。
:::

### 文件过滤

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **最小文件大小** | `minFilesize` | 单位 MB，小于此大小的视频文件跳过（过滤样片/预告片），`0` = 不限制 |
| **黑名单关键词** | `blacklistWords` | 文件名包含这些关键词的文件在扫描时**自动从源目录删除**（如 `["sample", "trailer"]`） |
| **视频扩展名** | `videoExts` | 自定义视频文件扩展名（留空则使用系统全局设置） |
| **元数据扩展名** | `metadataExts` | 自定义元数据文件扩展名（留空则使用系统全局设置） |

::: danger 注意
`blacklistWords` 匹配到的文件会被**直接删除**，请谨慎设置。
:::

### 命名与分类

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **命名规则** | `ruleId` | 选择一个命名规则（ID），决定整理后的文件夹和文件名格式。详见[规则引擎 - 命名规则](/features/rules#命名规则) |
| **启用分类引擎** | `categoryEnabled` | 开启后根据 TMDB 元数据自动分流到不同子目录（如动漫/国漫、电影/华语电影）。详见[规则引擎 - 分类规则](/features/rules#分类规则) |
| **覆盖已有** | `overwrite` | 目标已存在同名文件时是否覆盖 |

::: warning 注意
自动整理会按方案配置真实移动、复制或链接文件，没有「试运行」开关。首次配置建议使用只包含少量文件的测试源目录，确认结果正确后再切换到正式目录。
:::

### 并发控制

| 配置项 | 字段 | 默认值 | 说明 |
|--------|------|--------|------|
| **并发数** | `concurrency` | 1 | 同时处理的文件数量。提高可加速但增加 API 调用压力 |
| **历史记录上限** | `historyLimit` | 10000 | 每个方案保留的整理记录数量，`0` = 使用默认值 |

### 元数据提取 (ffprobe)

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **源文件探测** | `extractMetadata` | 对源文件执行 ffprobe 探测，获取精确的编码、分辨率等信息用于命名 |
| **目标文件探测** | `destExtractMetadata` | 对目标已有文件执行 ffprobe 探测，用于洗版升级的质量对比 |
| **缓存探测结果** | `cacheMetadata` | 在内存中缓存 ffprobe 结果，避免同一任务中重复探测 |
| **复用同系列结果** | `probeCache` | 同系列 + 同发布组的文件复用探测结果（如同组的全集共享编码信息） |

::: tip 建议
如果使用「标准命名」等不需要编码信息的命名规则，不需要开启 ffprobe 探测。仅在使用「详细命名」、「发烧友命名」等需要编码信息的规则时，或者开启洗版升级时，才需要开启。
:::

## 定时任务

支持 Cron 表达式定时执行整理：

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **Cron 表达式** | `cronExpr` | 标准 5 位 Cron 表达式 |
| **启用定时** | `cronEnabled` | 开关定时任务 |

```
0 */6 * * *    # 每 6 小时执行一次
0 2 * * *      # 每天凌晨 2 点执行
*/30 * * * *   # 每 30 分钟执行一次
```

## 实时整理

除定时任务外，还支持实时触发方式，新文件到达时自动开始整理。

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **实时模式** | `realtimeMode` | `off`（关闭）/ `listener`（文件监听）/ `polling`（轮询） |
| **延迟时间** | `realtimeDelay` | 单位秒，检测到新文件后等待多久再开始整理（等待文件下载完成） |

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `off` | 关闭实时整理 | 仅使用定时任务或手动触发 |
| `listener` | 通过 CD2 文件变化事件触发 | CD2 支持事件通知时 |
| `polling` | 定期扫描源目录是否有新文件 | 通用方案，兼容性最好 |

## 洗版升级

开启洗版后，当目标目录已有同一部影片时，系统会先检查候选资源和现有资源是否满足评分规则中的硬性条件，再比较质量分数。

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **启用洗版** | `upgradeEnabled` | 开启/关闭洗版升级功能 |
| **删除旧文件** | `upgradeDeleteOld` | 升级时删除旧文件（`false` 则移到回收目录） |
| **评分规则** | `qualityRuleId` | 选择一个质量评分规则（ID）。详见[规则引擎 - 质量评分规则](/features/rules#质量评分规则) |

![按分类目录选择评分规则示例](/screenshots/organize-quality-routing.png)

*通常先选择一条默认评分规则；不同分类需要不同偏好时，再开启“按分类目录选择评分规则”并逐项覆盖。未单独设置的目录继续继承默认规则。*

### 资格优先于总分

可以在 `audio_codec` 和 `subtitle` 评分项中设置：

- `required`：至少命中一个音轨/字幕轨优先条件。
- `min_tracks` / `max_tracks`：全部可用轨道数量硬限制。
- `preferred_tracks`：只在同分、同优先级时偏好更接近理想数量的资源。
- `priority` 轨道条件：按同一条轨道的语言、编码、声道、Atmos、字幕来源等组合匹配。

洗版决策遵循以下规则：

1. 合格资源始终优先于不合格资源，不受总分影响。
2. 候选资源不合格时不会替换现有资源。
3. 候选资源和现有资源都不合格时保留现有资源。
4. 两者都合格时先比较总分，并遵守设置的最低升级分差。
5. 总分完全相同时，才使用轨道优先级和 `preferred_tracks` 破局。
6. 同一集存在多个旧候选时，新文件会与每一个候选分别确定可比维度并评分，必须全部胜出才会替换。

首次整理时，如果启用了轨道硬性条件但当前候选不合格，系统会跳过该候选，不会先整理入库再等待洗版。

没取到信息时，系统会读取对应评分维度的 `unknown_policy`：默认的 `ignore` 会让双方跳过该维度，`zero` 会让未知侧记 0 分、已知侧正常评分。完整 FF 明确没有有效视频、完整轨道清单为空等“确认不存在”仍作为已知的 0 分参与；无 HDR 标签继续按产品规则视为 SDR。完整口径见[缺少信息时到底怎么评分](/features/rules#缺少信息时到底怎么评分)。

### 查看洗版依据

评分工具和整理记录使用同一套对比视图，可以直接看到：

- 双方是否合格、全部拒绝原因，以及最终由资格、加权总分、轨道优先级还是目标轨数距离决胜
- 各维度检测值、实际贡献和理论满分；`ignore` 跳过的维度会显示原因，`zero` 下的未知值显示“未知 / 0.00”
- 命中的内封或外挂音轨/字幕、真实轨道数量、`priority` 等级和目标轨数距离
- 本次实际使用的评分规则 ID 和名称，包括分类目录覆盖后的规则

成功洗版后，记录会保存当时双方的评分结果、实际规则和决胜依据。以后修改评分规则不会改写这次历史判断；旧记录缺少新字段时仍按已有分数兼容展示。

![整理记录中的洗版评分详情](/screenshots/organize-record-score-detail.png)

*在成功记录中展开条目，再点击文件旁的“洗版”标签，即可查看当时使用的规则、双方得分和被忽略的维度。*

### 外挂字幕参与判断

外挂字幕不是独立整理对象。系统先以**当前目录**为范围建立视频与字幕的关联关系，再把成功关联的字幕用于评分和伴随文件转移。不会扫描字幕子目录，也不会跨目录寻找视频。

#### 后缀是否启用

字幕是否进入整理流程，首先取决于当前任务最终生效的 `metadataExts`：方案未自定义时使用全局配置，方案已自定义时使用方案配置。只有同时满足“后缀已配置”和“系统支持对应处理能力”的文件才会进入后续步骤。

| 后缀 | 转移 | 字幕评分 | 说明 |
|------|------|----------|------|
| `.srt` / `.ass` / `.ssa` / `.sub` / `.sup` | 是 | 是 | 配置启用并成功关联后参与评分 |
| `.vtt` | 是 | 是 | 默认元数据后缀不包含 `.vtt`，需要用户主动加入 |
| `.idx` | 跟随 `.sub` | 否 | 只有同名 `.sub` 已成功关联时才跟随转移，不重复计算字幕轨 |
| `.smi` / `.ttml` | 是 | 否 | 用户配置后可作为伴随文件转移，暂不进入字幕评分 |
| `.nfo` / `.jpg` / `.json` 等 | 是 | 否 | 按普通元数据规则处理，不作为字幕 |

从 `metadataExts` 移除某个后缀后，该后缀会同时退出字幕关联、评分和伴随转移；不会因为它属于常见字幕格式而被自动加回。

#### 关联顺序

每个字幕按以下顺序寻找同目录视频，只有当前最高匹配层级得到**唯一候选**时才认领。一个字幕最多归属一个视频。

1. **完整文件名匹配**：视频 stem 与字幕 stem 相同，或字幕仅在视频 stem 后增加语言、`forced` 等字幕描述。
2. **规范化文件名匹配**：统一全半角、大小写、标点和连续分隔符后相同。
3. **文件名媒体身份匹配**：分别从视频和字幕文件名提取标题、年份、季、集，再比较结构化身份；不使用编辑距离做相似标题兜底。

```text
Show.S01E03.1080p.WEB-DL.mkv
Show.S01E03.1080p.WEB-DL.zh-CN.ass  -> 完整文件名匹配

Show.Name-S01E03.mkv
show_name_S01E03.zh-CN.srt          -> 规范化文件名匹配

Show.S01E03.1080p.WEB-DL.x265.mkv
Show.S01E03.zh-CN.ass                -> 文件名媒体身份匹配
```

剧集关联还遵守以下限制：

- 双方都解析出季、集时，季号和集号必须一致；冲突时直接拒绝。
- `S01E03.chs.ass` 没写剧名时，只能关联目录内唯一的 `S01E03` 视频。
- `E03.chs.ass` 只有集号时，仅当当前目录所有可识别视频都属于同一季，才会推断季号。
- 字幕解析出剧名时，必须与视频的任一标题一致。
- 同一季集存在多个视频版本，而字幕名不足以区分版本时，判定为歧义。

电影关联还遵守以下限制：

- 字幕标题必须与视频的任一标题一致。
- 双方都有年份时，年份必须一致。
- 字幕没有年份时，只有当前目录存在唯一同标题电影才接受。
- 只有年份、没有有效标题时不关联，也不会因为目录只有一个视频就强制归属。

#### 评分、转移和安全处理

成功关联的可评分字幕会作为 `external` 字幕轨进入来源去重、首次整理资格判断和目标洗版评分。没有语言标记时语言记为 `und`，会计入字幕数量，但不能满足明确的中文条件。

伴随转移时，字幕统一重命名为“目标视频 stem + 原字幕语言/属性后缀 + 字幕扩展名”：

```text
S01E03.zh-CN.ass
-> 剧名 - S01E03 - 2160p.zh-CN.ass
```

以下字幕不会参与评分、转移、重命名、洗版删除或源文件删除：

- 当前目录没有视频；
- 找不到匹配视频；
- 季、集、标题或年份冲突；
- 同一匹配层级得到多个候选，无法唯一归属。

这类字幕会标记为本次任务的受保护文件，阻止空目录清理误删，并在日志中记录字幕身份、候选视频和拒绝或歧义原因。未识别主视频没有可靠媒体身份时，只使用原有完整文件名规则，不启用宽松关联。

常见结果：

```text
Movie.mkv + Movie.zh-CN.srt          -> 可靠关联，语言 zh-CN
Movie.mkv + Movie.srt                -> 可靠关联，语言 und
Show.S01E03.mkv + Show.S01E03.chs.ass -> 可靠关联，语言 zh-CN
Show.S01E03.1080p.mkv
Show.S01E03.2160p.mkv
S01E03.zh-CN.ass                      -> 候选不唯一，保护字幕但不处理
Movie.mkv + MovieExtra.srt            -> 身份不匹配，不处理
```

完整评分字段、格式兼容值和语言归一化示例请参阅[规则引擎 - 质量评分规则](/features/rules#质量评分规则)。

## Emby 刷新联动

整理完成后可自动触发 Emby 媒体库刷新：

| 配置项 | 字段 | 说明 |
|--------|------|------|
| **启用 Emby 刷新** | `embyRefreshEnabled` | 整理完成后自动刷新 Emby 媒体库 |
| **刷新服务器** | `embyRefreshServers` | 指定刷新哪些 Emby 服务器（留空 = 全部） |
| **刷新延迟** | `embyRefreshDelay` | 整理完成后等待多少秒再刷新（`0` = 立即） |
| **路径映射** | `embyPathMappings` | 本地路径 → Emby 路径的映射（Docker 环境中路径可能不同） |

路径映射示例：

```json
[
  {
    "localPrefix": "/mnt/media",
    "embyPrefix": "/media"
  }
]
```

当 MediaTidy 的文件路径是 `/mnt/media/电影/阿凡达`，Emby 中对应路径是 `/media/电影/阿凡达` 时，需要配置此映射。

## 整理记录

「整理记录」页面分为**成功记录**和**失败记录**两个页签。成功记录用于追溯已入库媒体和洗版判断；失败记录用于继续处理已经移入未识别目录的文件。

### 成功记录

每次成功整理的文件都会被记录，并按 TMDB 条目分组展示：

- **分组视图**：显示海报、标题、年份、媒体类型、文件数量和最新整理时间
- **关键词搜索**：按标题模糊搜索
- **明细查看**：点击某个 TMDB 条目，展开查看该条目下所有文件记录

#### 单条记录字段

| 字段 | 说明 |
|------|------|
| `srcPath` | 原始文件路径（整理前的位置） |
| `destPath` | 目标文件路径（整理后的位置） |
| `destName` | 目标文件名 |
| `transferType` | 传输方式（move / copy / symlink / hardlink） |
| `tmdbId` | 匹配的 TMDB ID |
| `mediaType` | movie 或 tv |
| `title` | 媒体标题 |
| `year` | 年份 |
| `season` / `episode` | 季/集号（仅剧集） |
| `posterUrl` | 海报地址 |
| `status` | `done`（已完成）/ `restored`（已还原）/ `missing`（文件丢失）/ `superseded`（已被后续记录替代） |
| `upgraded` | 是否为洗版升级的结果 |
| `oldFileName` | 洗版时被替换的旧文件名 |
| `newScore` / `oldScore` | 洗版时双方完整的评分快照 |
| `qualityRuleId` / `qualityRuleName` | 本次实际生效的评分规则 ID 和名称 |
| `upgradeBasis` / `upgradeDimension` | 洗版决胜类型和对应的轨道维度 |

洗版记录可展开统一评分视图，查看资格、总分、各维度贡献、轨道偏好和最终决胜依据。分类目录覆盖评分规则时，这里显示覆盖后真正使用的规则。

#### 还原文件

选择一条或多条 `done` 状态的记录，点击「还原」：

- 将文件从目标路径移回源路径（或指定目标目录）
- 还原后记录状态变为 `restored`
- 如果目标文件已不存在，记录自动标记为 `missing`

#### 重新识别

当文件被匹配到错误的 TMDB 条目时：

1. 选择需要纠正的记录
2. 指定正确的 TMDB ID 和媒体类型
3. 系统自动重新命名并移动文件到正确位置
4. 记录中的元数据同步更新

#### 删除记录

- 删除选中的记录（仅删除数据库记录，不影响实际文件）
- 也可以删除某个方案下的全部记录

### 失败记录

自动整理把未识别文件**成功移动到未识别目录后**，会建立独立失败记录，保存原路径、当前位置和结构化失败原因。仅跳过文件或移动失败时不会建立这类记录。

![整理失败记录列表与操作入口](/screenshots/organize-failure-records.png)

*先用方案、状态或关键词筛选记录；勾选待处理文件后可以批量手动整理或删除，也可以使用“扫描存量”导入未识别目录中的旧文件。*

| 状态 | 含义 |
|------|------|
| `pending` | 待处理，文件仍在未识别目录，可继续整理或删除 |
| `organized` | 已通过失败记录重新整理成功，并已写入成功记录 |
| `deleted` | 已从未识别目录删除媒体文件及伴随文件 |
| `missing` | 处理时发现当前位置的文件已经不存在 |

失败记录支持按方案、状态和关键词筛选，并提供以下操作：

- **单条或批量手动整理**：选择正确的 TMDB 条目和媒体类型后重新提交整理；成功时状态变为 `organized`，同时生成正常成功记录并清理未识别目录内的空父目录
- **删除文件**：永久删除选中的媒体文件及同名字幕、NFO 等伴随文件；操作范围会校验在当前方案的未识别根目录内，不能借记录删除目录外文件
- **扫描存量**：按方案扫描现有未识别目录，把此前已经存在的视频去重导入失败记录；这类历史导入记录可能没有原始来源路径

::: warning 删除失败文件
“删除文件”会删除实际媒体和伴随文件，不只是删除数据库记录，而且无法撤销。执行前请核对当前路径。
:::

### 历史记录上限

| 配置项 | 字段 | 默认值 | 说明 |
|--------|------|--------|------|
| **记录上限** | `historyLimit` | `10000` | 每个方案分别保留成功记录和已处理失败记录，超出后清理各自最早记录 |

仍为 `pending` 的失败记录不会因历史上限而被清理，避免尚未处理的文件从列表中消失。

## 手动整理

在「文件管理」页面选择文件后，可以手动触发整理：

1. 选择一个或多个文件
2. 点击「手动整理」
3. 选择一个已有的整理方案
4. 手动指定 TMDB ID 和媒体类型（movie / tv）
5. 系统按方案配置的命名规则执行转移

手动整理的结果同样记录在整理记录中，可以后续进行还原或重新识别。

## 伴随文件

整理主视频文件时，当前 `metadataExts` 允许且已关联到该视频的伴随文件会一起转移：

- **字幕**：`.srt`、`.ass`、`.ssa`、`.sub`、`.sup`；配置后也支持 `.vtt`。`.idx` 仅随已关联的同名 `.sub` 转移。
- **仅转移字幕格式**：`.smi`、`.ttml` 可按配置转移，暂不评分。
- **元数据**：例如 `.nfo`、`.txt`、`.json`。
- **图片**：例如 `.jpg`、`.jpeg`、`.png`、`.tbn`。

普通元数据和图片仍按严格伴随文件名关联；字幕使用上文的[目录级字幕关联](#外挂字幕参与判断)。没有同目录主视频或无法唯一归属的视频字幕，不会被当作独立文件整理。
